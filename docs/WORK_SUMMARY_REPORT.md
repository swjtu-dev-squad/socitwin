## 2. 关键成果与交付物

### 2.1. Issue #13 修复：日志增量语义错误

本轮最重要的成果是彻底解决了 Issue #13 中描述的日志重复问题。

*   **问题现象**：在多步仿真中，前端日志流会重复显示历史帖子，重复率随步数增加而升高，在第 5 步达到 50%。
*   **根因定位**：后端引擎 `real_oasis_engine_v3.py` 中的 `_read_posts_table()` 和 `_read_interactions_table()` 方法每次都从数据库读取全量历史记录，而非增量新记录。
*   **修复方案**：引入了 **“水位线游标（Watermark Cursor）”** 机制。
    1.  在引擎类中增加 `_last_seen_post_id` 和 `_last_seen_like_id` 状态变量。
    2.  修改日志读取的 SQL 查询，增加 `WHERE post_id > ?` 条件，并在每次读取后更新水位线。
    3.  在仿真初始化（`initialize`）和重置（`reset`）时，将水位线归零，确保会话隔离。

## 3. 遗留观察与后续建议

在测试过程中，发现两个与本次修复无关但值得关注的现象：

1.  **仿真“空跑”现象**：在最小测试配置（1 agent, 5 steps）下，仿真引擎的 `step` 方法耗时极短（<1s），且数据库中未产生新的帖子（`totalPosts=0`）。这表明 LLM 的决策与行动链路未被有效触发。**建议**：在下一阶段，应在更大规模（如 ≥3 agents, ≥10 steps）的配置下进行扩展测试，以验证智能体的真实行为产出。

2.  **SQLite 并发写入冲突**：当多个进程（如 `pnpm dev` 服务和独立的 Python 脚本）同时尝试写入 `oasis_simulation.db` 文件时，会触发 `disk I/O error`。**建议**：在项目文档中明确指出，任何与仿真数据库的交互都应通过 API 服务进行，避免直接操作数据库文件，确保单进程写入原则。
